rakudo:

13:07:48 <diakopter> rakudo: sub foo(:$w=4){say $w};foo('w'=>3);foo(w=>3);
13:07:49 <?? exp_evalbot ??> OUTPUT[3␤3␤]

Note pugs' behavior:

13:07:56 <diakopter> pugs: sub foo(:$w=4){say $w};foo('w'=>3);foo(w=>3);
13:07:57 <?? exp_evalbot ??> OUTPUT[4␤3␤]


S06 says "Pairs intended as positional arguments rather than named
arguments may be indicated by extra parens or by explicitly quoting
the key to suppress autoquoting" and:

Only bare keys with valid identifier names are recognized as named arguments:

doit when => 'now'; # always a named arg
doit 'when' => 'now'; # always a positional arg

Please be _very_careful_ when using evalbot to report rakudo bugs.
At the time that the above was done, evalbot was running a version
of rakudo that was quite a bit older -- at least ten hours -- than
the version in trunk. The version of rakudo that was in the trunk
at the time of the above test actually reports:

$ ./parrot perl6.pbc
> sub foo(:$w=4){say $w};foo('w'=>3);foo(w=>3);
w 3
3

While this version of rakudo also gives incorrect output, it does
show that if evalbot is going to be used to report tests, we need
some verification or independent test against rakudo's trunk.

I think the above incorrect output is due to a bug in Parrot's
calling conventions, but will check on that and file a Parrot
ticket if so.

> Note pugs' behavior:
>
> 13:07:56 <diakopter> pugs: sub foo(:$w=4){say $w};foo('w'=>3);foo(w=>3);
> 13:07:57 <?? exp_evalbot ??> OUTPUT[4␤3␤]

I'm going to argue that pugs is incorrect here. The call
foo('w'=>3) has a positional parameter that doesn't match
the signature of foo.
