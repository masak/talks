Rakudo as of r28692 reports a slightly wrong context on reporting parse
failures.

When trying to parse t/spec/S02-builtin_data_types/lists.t (r21012) it
reports:

Statement not terminated properly at line 118, near "= \\@array;"
current instr.: 'parrot;PGE::Util;die' pc 120
(runtime/parrot/library/PGE/Util.pir:82)

But line 118 reads like this:
my $arrayref = \@array;

So there's just a single backslash, while two get reported. I see no
need for extra quoting in the error message.

> I think a better option is probably to escape only non-printables,

I don't know if you're including newline as a "non-printable".
If we treat newline as printable, then our error message would end
up like:

Statement not terminated properly at line 188, near "=
\@array;"

(In this specific instance, where prefix:<\\> is syntactically correct
but Rakudo doesn't recognize it yet, we probably should be failing with
"prefix:<\\> not implemented" instead of a syntax error. I'll probably
add that later today. But the question of how we should quote context in
the case of a true syntax error still remains.)
