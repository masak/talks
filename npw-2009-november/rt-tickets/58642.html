By default Range (and probably many other classes) constructed in
non-consistent state.

Minimal example:
.namespace

load_bytecode "perl6.pbc"
$P0 = new 'Range'
$P1 = clone $P0
say "done"
.end

(src/gen_builtins.pir:415)

be using the 'new' opcode to create objects. The correct
approach is

## $P1 = Range.new;

Ouch, yes. That was a really nasty one to track down. Basically, we
never tend to actually construct a list like that (normally we do it
from slupys, which means we've really got a ResizablePMCArray that is
mapped through various magic), though it is of course valid to say
List.new in Perl 6. And when we called .new, it went through all
attributes of the class and initialized them, either with values passed
to new or with Undef. It was then ending up overwriting the instances of
the PMCs that we keep so we can delegate to them in PMC inheritance with
Undef, and thus the error.

Anyway, resolved (I hope!) in r31005.
