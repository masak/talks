Hi,

when playing with the stuff in 'languages/perl6', I noticed that code like
print( +42, "\n" );
didn't do the right thing. 'perl6' printed '42' but forgot about the "\n".

The cause seemed to me that '+' isn't set up as an unary operator in
'P6C/Parser.pm'.
After adding '+' in 'Parser.pm' and in a couple of othe files, it now seems
to behave as expected.

On my Linux machine there are some other tests failing in t/compiler, but
these seem not to be related to unary '+'.
One of these failures is a 'inf' vs. 'Inf' issue.

A patch is attached.

Looks good, except that this needs to make sure an int is being
returned, e.g.

+"42" -> 42
+"forty-two" -> 0

The lazy man in me would just shove it through an int reg, but that
loses precision if we go to bignums. Though for the moment I can't
think of a better way.

/s

+# unary plus.
+sub prefix_pos {

+ my $tmp = $x->args->val;

+END
+ return scalar_in_context($res, $x->{ctx});

Sean pointed out that
> +"42" -> 42
> +"forty-two" -> 0

Catching up on 'Apacolypse 3' and 'Exegeses 3' left me partly
enlightened and partly confused. Unary '+' should put the left into
numeric context. This should turn +"forty-two" into NaN.
Also it says that unary '-' is left as in Perl5. Does this meant that in
Perl6 we have:
-"--opt" -> "+-opt"
-"+-opt" -> "++-opt" ?
This seems kind of baroque for Perl6.

In order to get at least
+"42_is_forty-two" -> 42
working I did a double negation in 'sub prefix_pos' in
'P6C/IMCC/prefix.pm'. This is a nasty workaround, but PIR and PASM don't
seem to have a 'pos' function.
A better solution might be a explicit 'to_num' call in the PIR-code.

A patch for 'IMCC/prefix.pm' and a couple test cases for
'perl6/t/builtins/strings.t' are included.
